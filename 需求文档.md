# 电商后台管理系统 - 需求文档

## 项目概述

### 1.1 项目背景
为配合毕业设计项目“基于Selenium和Python的电商后台管理系统自动化测试”，需要开发一个完整的电商后台管理系统作为测试目标。该系统将作为自动化测试框架的验证对象，模拟真实电商后台的核心业务场景。

### 1.2 项目目标
1. 提供完整的电商后台管理功能，支持自动化测试验证
2. 实现标准RESTful API接口，支持前后端分离架构
3. 确保页面元素结构与自动化测试框架的定位器完全兼容
4. 提供可配置的测试数据初始化功能
5. 支持多环境部署（开发、测试、演示）

### 1.3 项目范围
- **后台管理系统**（核心）：管理员登录、商品管理、订单管理、用户管理、权限管理
- **API接口**：完整的RESTful API，支持前后端分离
- **数据库**：MySQL数据库，包含核心业务数据表
- **管理界面**：基于Web的后台管理界面

### 1.4 与自动化测试框架的兼容性要求
本系统的页面元素ID、class名称必须与自动化测试框架中的定位器完全匹配：
- 登录页面：`username`、`password`、`login-btn`、`error-msg`、`remember-me`
- 其他页面：按照自动化测试框架的页面对象模型定义

## 技术架构

### 4.1 架构设计原则
1. **前后端分离**：前端独立部署，通过API与后端交互
2. **模块化设计**：功能模块高内聚、低耦合
3. **RESTful API**：遵循RESTful设计规范
4. **响应式设计**：支持不同设备和分辨率
5. **安全性优先**：多层次安全防护

### 4.2 技术栈选择（推荐方案）

#### 方案一：Spring Boot + Vue.js（推荐）
**后端：**
- 框架：Spring Boot 2.7+
- 安全：Spring Security + JWT
- 数据库：MySQL 8.0
- ORM：MyBatis-Plus
- 缓存：Redis（可选）
- 文档：Swagger/OpenAPI 3.0

**前端：**
- 框架：Vue 3 + Composition API
- UI库：Element Plus
- 路由：Vue Router 4
- 状态管理：Pinia
- 构建工具：Vite

**优点：** 技术成熟、社区活跃、适合Java技术栈

#### 方案二：Node.js + Vue.js
**后端：**
- 框架：Express.js 4.x 或 NestJS
- 数据库：MySQL 8.0 + Sequelize/TypeORM
- 安全：JWT、bcrypt、helmet
- 验证：Joi或class-validator

**前端：** 同方案一

**优点：** 全JavaScript技术栈、开发效率高

#### 方案三：Django + Vue.js
**后端：**
- 框架：Django 4.x + Django REST Framework
- 数据库：PostgreSQL或MySQL
- 安全：Django内置安全机制

**前端：** 同方案一

**优点：** Python技术栈、Admin后台强大

## 数据库设计

### 5.1 核心数据表

#### 用户表 (user)
```sql
CREATE TABLE user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '用户ID',
    username VARCHAR(50) UNIQUE NOT NULL COMMENT '用户名',
    password VARCHAR(255) NOT NULL COMMENT '加密密码',
    email VARCHAR(100) UNIQUE COMMENT '邮箱',
    phone VARCHAR(20) COMMENT '手机号',
    avatar VARCHAR(255) COMMENT '头像',
    status TINYINT DEFAULT 1 COMMENT '状态: 0-禁用, 1-正常',
    user_type TINYINT DEFAULT 1 COMMENT '用户类型: 1-普通用户, 2-管理员',
    last_login_time DATETIME COMMENT '最后登录时间',
    login_fail_count INT DEFAULT 0 COMMENT '登录失败次数',
    lock_time DATETIME COMMENT '锁定时间',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_username (username),
    INDEX idx_email (email),
    INDEX idx_status (status)
) COMMENT '用户表';
```

#### 角色表 (role)
```sql
CREATE TABLE role (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '角色ID',
    name VARCHAR(50) UNIQUE NOT NULL COMMENT '角色名称',
    code VARCHAR(50) UNIQUE NOT NULL COMMENT '角色编码',
    description VARCHAR(255) COMMENT '角色描述',
    status TINYINT DEFAULT 1 COMMENT '状态: 0-禁用, 1-正常',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
) COMMENT '角色表';
```

#### 权限表 (permission)
```sql
CREATE TABLE permission (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '权限ID',
    name VARCHAR(50) NOT NULL COMMENT '权限名称',
    code VARCHAR(100) UNIQUE NOT NULL COMMENT '权限编码',
    type TINYINT NOT NULL COMMENT '权限类型: 1-菜单, 2-操作',
    parent_id BIGINT DEFAULT 0 COMMENT '父权限ID',
    path VARCHAR(255) COMMENT '菜单路径',
    icon VARCHAR(50) COMMENT '菜单图标',
    sort_order INT DEFAULT 0 COMMENT '排序',
    status TINYINT DEFAULT 1 COMMENT '状态: 0-禁用, 1-正常',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
) COMMENT '权限表';
```

#### 商品表 (product)
```sql
CREATE TABLE product (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '商品ID',
    name VARCHAR(200) NOT NULL COMMENT '商品名称',
    description TEXT COMMENT '商品描述',
    price DECIMAL(10,2) NOT NULL COMMENT '商品价格',
    status TINYINT DEFAULT 1 COMMENT '状态: 0-下架, 1-上架',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_status (status),
    INDEX idx_price (price)
) COMMENT '商品表';
```

#### 订单表 (order)
```sql
CREATE TABLE `order` (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '订单ID',
    order_no VARCHAR(50) UNIQUE NOT NULL COMMENT '订单编号',
    user_id BIGINT NOT NULL COMMENT '用户ID',
    total_amount DECIMAL(10,2) NOT NULL COMMENT '订单总金额',
    status TINYINT NOT NULL COMMENT '订单状态: 1-待付款, 2-待发货, 3-已发货, 4-已完成, 5-已取消',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_user (user_id),
    INDEX idx_status (status),
    INDEX idx_order_no (order_no)
) COMMENT '订单表';
```

## API设计

### 6.1 API规范
- 协议：HTTP/HTTPS
- 数据格式：JSON
- 编码：UTF-8
- 认证：JWT Token（Authorization: Bearer <token>）
- 版本：v1（路径前缀：/api/v1/）

### 6.2 核心API接口

#### 认证相关
```
POST   /api/v1/auth/login          # 用户登录
POST   /api/v1/auth/logout         # 用户注销
POST   /api/v1/auth/refresh-token  # 刷新Token
GET    /api/v1/auth/user-info      # 获取用户信息
```

#### 商品管理
```
GET    /api/v1/admin/products      # 商品列表（分页、搜索）
GET    /api/v1/admin/products/{id} # 商品详情
POST   /api/v1/admin/products      # 添加商品
PUT    /api/v1/admin/products/{id} # 更新商品
DELETE /api/v1/admin/products/{id} # 删除商品
PATCH  /api/v1/admin/products/{id}/status # 更新状态
POST   /api/v1/admin/products/upload-image # 上传图片
```

#### 订单管理
```
GET    /api/v1/admin/orders        # 订单列表
GET    /api/v1/admin/orders/{id}   # 订单详情
PUT    /api/v1/admin/orders/{id}/status # 更新订单状态
GET    /api/v1/admin/orders/export # 导出订单
GET    /api/v1/admin/orders/statistics # 订单统计
```

#### 用户管理
```
GET    /api/v1/admin/users         # 用户列表
GET    /api/v1/admin/users/{id}    # 用户详情
POST   /api/v1/admin/users         # 添加用户
PUT    /api/v1/admin/users/{id}    # 更新用户
PATCH  /api/v1/admin/users/{id}/status # 更新用户状态
GET    /api/v1/admin/users/{id}/orders # 用户订单
```

#### 权限管理
```
GET    /api/v1/admin/roles         # 角色列表
POST   /api/v1/admin/roles         # 添加角色
PUT    /api/v1/admin/roles/{id}    # 更新角色
DELETE /api/v1/admin/roles/{id}    # 删除角色
GET    /api/v1/admin/permissions   # 权限列表
PUT    /api/v1/admin/roles/{id}/permissions # 分配权限
```

### 7.2 前端页面元素规范
为确保与自动化测试框架兼容，必须遵守以下规范：

#### 通用规范
1. 所有表单元素必须有唯一的`id`属性
2. 操作按钮必须有明确的`id`或易于定位的`class`
3. 错误提示使用统一class：`error-msg`
4. 成功提示使用统一class：`success-msg`
5. 加载状态使用统一class：`loading`

#### 表格规范
```html
<table id="[模块名]-table" class="data-table">
  <thead>
    <tr>
      <th>ID</th>
      <th>名称</th>
      <th>操作</th>
    </tr>
  </thead>
  <tbody>
```

## AI智能运营助手需求补充

### A.1 功能定位
AI智能运营助手面向后台管理员、商家和运营人员提供销售分析、补货建议、促销建议与异常提醒能力。功能以“智能问讯窗口 + 今日建议卡片”形式嵌入后台仪表盘，基于最近售出的商品、订单统计与库存状态生成建议。

### A.2 建设目标
1. 提供基于最近7天、15天、30天销售数据的智能问答能力。
2. 帮助运营人员快速识别热销商品、低库存商品、异常订单和促销机会。
3. 在不影响现有订单、商品、权限模块的前提下，新增可扩展的智能分析能力。
4. 第一阶段优先实现数据驱动建议，第二阶段支持接入大模型实现自然语言增强。

### A.3 适用角色
| 角色 | 数据范围 | 可用能力 |
|------|----------|----------|
| 系统管理员 | 全量数据 | 查看全部经营建议、异常预警、趋势分析 |
| 商家 | 自有商品与相关订单 | 查看本店热销商品、补货建议、促销建议 |
| 运营人员 | 授权范围数据 | 查看经营建议、销售趋势、风险提醒 |

### A.4 功能范围

#### A.4.1 智能问讯窗口
1. 在仪表盘提供固定入口，支持悬浮按钮、侧边抽屉或卡片式问答区。
2. 支持快捷问题：
   - 最近什么商品卖得最好
   - 哪些商品需要优先补货
   - 哪些商品适合做促销
   - 最近订单有什么异常
   - 给我一段今日运营建议
3. 支持按时间范围切换：最近7天、最近15天、最近30天。
4. 问答结果需展示生成依据，包括销量、销售额、库存、订单状态数量等数据。

#### A.4.2 热销商品建议
1. 基于已完成订单、订单明细、商品销量生成热销排行。
2. 输出商品名称、销量、销售额、库存状态、建议动作。
3. 当某商品连续处于热销状态时，提示优先补货或增加推荐位。

#### A.4.3 智能补货建议
1. 根据最近销售速度与当前库存计算缺货风险。
2. 输出建议补货商品、建议补货优先级、风险等级。
3. 可设置简单库存安全阈值：
   - 高风险：库存低于最近7天销量
   - 中风险：库存低于最近15天平均销量
   - 低风险：库存充足

#### A.4.4 促销建议
1. 对销量一般但库存偏高的商品给出促销建议。
2. 对高转化、高客单价商品给出重点推荐建议。
3. 输出建议方式，例如首页推荐、限时折扣、组合促销。

#### A.4.5 异常订单提醒
1. 识别待付款订单偏多、待发货积压、取消订单增多等情况。
2. 通过仪表盘建议卡片或问讯窗口输出异常摘要。
3. 提示对应处理动作，例如优先发货、检查价格策略、关注售后问题。

#### A.4.6 今日运营摘要
1. 仪表盘展示一段简明的今日经营建议。
2. 内容包括今日订单概览、热销商品提醒、补货提醒、风险提醒。
3. 支持点击跳转到订单管理、商品管理相关页面。

### A.5 数据来源
AI智能运营助手一期仅使用系统内部业务数据，不依赖外部平台数据。主要来源如下：
1. 订单表 `order`
2. 订单明细表 `order_item`
3. 商品表 `product`
4. 用户与权限范围数据

### A.6 核心规则
1. 销售统计默认基于已完成订单统计。
2. 热销商品按照销量优先排序，销售额作为辅助排序依据。
3. 补货建议由最近销量、当前库存和安全阈值共同决定。
4. 促销建议由库存偏高且近期销量偏低规则触发。
5. 异常提醒由订单状态结构变化触发，例如待发货占比异常升高。

### A.7 交互要求
1. 问讯窗口回答应简洁、结构化，优先输出结论和行动建议。
2. 每条建议需显示生成依据，避免仅给出模糊结论。
3. 建议内容应支持“查看详情”跳转至订单管理或商品管理页面。
4. 当数据不足时，系统应提示当前时间范围内数据不足，无法生成可靠建议。

### A.8 非功能要求
1. 问讯接口平均响应时间不高于2秒。
2. 首屏建议卡片加载应与仪表盘同时完成或在3秒内完成。
3. 角色隔离必须生效，商家只能查看自身数据范围内建议。
4. 建议生成过程需保留日志，便于排查规则与结果是否一致。

### A.9 分阶段实施

#### A.9.1 第一阶段：数据驱动智能助手
1. 提供固定问题模板。
2. 提供规则引擎生成建议文案。
3. 提供热销商品、补货建议、异常提醒、运营摘要。
4. 不强依赖外部AI服务。

#### A.9.2 第二阶段：大模型增强
1. 接入大模型服务，实现自由提问与追问。
2. 支持自然语言总结与更灵活的运营建议生成。
3. 保留一期规则输出作为兜底结果，避免AI服务异常影响核心功能。
