# 电商后台管理系统 - 计划架构文档

## 1. 项目概述

### 1.1 项目背景
为配合毕业设计项目“基于Selenium和Python的电商后台管理系统自动化测试”，本项目旨在开发一个完整的电商后台管理系统作为测试目标。该系统将作为自动化测试框架的验证对象，模拟真实电商后台的核心业务场景，确保自动化测试框架的完整性和有效性。

### 1.2 项目目标
1. **测试验证目标**：提供完整的电商后台管理功能，支持自动化测试框架全面验证
2. **技术实现目标**：实现前后端分离架构，采用现代化技术栈
3. **兼容性目标**：确保页面元素结构与自动化测试框架定位器完全匹配
4. **可部署目标**：支持多环境部署，提供完整的测试数据初始化功能
5. **性能目标**：满足自动化测试对响应时间和并发性能的要求

### 1.3 项目范围
**包含模块：**
- **后台管理系统（核心）**：管理员登录、商品管理、订单管理、用户管理、权限管理
- **API接口层**：完整的RESTful API，支持前后端分离
- **数据库系统**：MySQL数据库，包含核心业务数据表
- **管理界面**：基于Web的后台管理界面，与测试框架完全兼容

**技术边界：**
- 前后端分离架构，不采用传统MVC模式
- 基于Web的后台管理系统，不包含移动端APP
- 支持主流浏览器兼容性（Chrome、Firefox、Edge）
- 提供完整的API文档和测试接口

### 1.4 与自动化测试框架的兼容性要求
**强制性兼容要求：**
1. **页面元素ID必须严格匹配**：所有表单元素、按钮、表格的ID必须与测试框架定义一致
2. **页面结构必须稳定**：页面DOM结构不能频繁变动，确保自动化测试稳定性
3. **错误处理必须规范**：错误提示使用统一的class和显示方式
4. **URL路径必须固定**：关键页面的访问路径必须保持不变

**关键元素映射表：**
| 页面模块 | 必须包含的元素ID | 测试框架定位器 |
|----------|------------------|----------------|
| 登录页面 | username, password, login-btn, remember-me, error-msg | LoginPage类定义 |
| 商品列表 | product-search, category-filter, product-table, add-product-btn | ProductListPage类定义 |
| 商品操作 | product-name, product-price, product-stock, save-product-btn | ProductOperationPage类定义 |
| 订单列表 | order-search-input, status-filter, order-table | OrderListPage类定义 |
| 用户列表 | user-search-input, user-table | UserListPage类定义 |

### 1.5 角色权限重构说明
为匹配最新需求分析，系统角色从“管理员 + 运营人员”的宽泛设计，收敛为以下三类核心角色，并以角色编码作为前后端统一权限依据：

| 角色 | 角色编码 | 能力范围 |
|------|----------|----------|
| 系统管理员 | `ROLE_ADMIN` | 全量商品、订单、用户、权限管理 |
| 商家 | `ROLE_MERCHANT` | 仅自己店铺商品、仅自己店铺订单 |
| 普通用户 | `ROLE_USER` | 商品浏览、本人订单、下单、取消订单 |

本次计划架构同时约束三个层面：
1. **菜单可见性**：不同角色只看到对应导航入口。
2. **接口访问性**：后端接口按角色放行，而不是只依赖前端隐藏按钮。
3. **数据范围**：商家和普通用户即使访问到了接口，也只能读取自己权限范围内的数据。

### 2.3 架构设计原则

#### 2.3.1 前后端分离原则
- **清晰的职责划分**：前端负责展示和交互，后端负责业务逻辑和数据
- **API契约优先**：先定义API接口，再并行开发
- **无状态设计**：后端API无状态，通过JWT进行身份验证

#### 2.3.2 模块化与微服务化（演进式）
- **当前阶段**：单体应用，按业务模块分包
- **演进方向**：支持未来拆分为微服务
- **模块边界**：用户、商品、订单、权限四大核心模块

#### 2.3.3 安全性设计原则
- **深度防御**：多层安全防护（网络层、应用层、数据层）
- **最小权限**：按角色分配最小必要权限
- **输入验证**：前端+后端双重验证
- **安全审计**：完整操作日志记录
- **数据范围隔离**：接口鉴权通过后，仍需对商家和普通用户追加数据级范围过滤

#### 2.3.4 可测试性设计
- **自动化测试友好**：页面元素ID固定，结构稳定
- **API测试支持**：完整的Swagger文档
- **测试数据管理**：支持测试数据初始化和清理
- **健康检查**：提供系统健康状态接口

#### 3.1.1.1 角色访问矩阵

| 资源 | 系统管理员 | 商家 | 普通用户 |
|------|------------|------|----------|
| 后台商品管理 `/api/admin/products/**` | 全量 | 仅自己创建/归属商品 | 不可访问 |
| 后台订单管理 `/api/admin/orders/**` | 全量 | 仅包含自己商品的订单 | 不可访问 |
| 后台用户管理 `/api/admin/users/**` | 全量 | 不可访问 | 不可访问 |
| 后台角色管理 `/api/admin/roles/**` | 全量 | 不可访问 | 不可访问 |
| 用户商品接口 `/api/products/**` | 可访问 | 可访问 | 可访问 |
| 用户订单接口 `/api/orders/**` | 可访问 | 不推荐使用 | 仅本人订单 |

实现约束：
- 管理员权限以角色编码 `ROLE_ADMIN` 为准。
- 商家权限以角色编码 `ROLE_MERCHANT` 为准，不再通过单一 `userType=2` 近似。
- 普通用户权限以 `ROLE_USER` 为基线角色。

#### 3.1.2 数据库设计要点
**核心表关系图：**
```
用户表(user) ──────┬─────── 用户角色表(user_role) ─────── 角色表(role)
                   │                                           │
                   │                                           │
商品表(product) ───┼─────── 订单商品表(order_item) ─────── 订单表(order)
                   │
                   │
分类表(category) ──┘
```

**关键设计决策：**
1. **软删除设计**：所有业务表添加`is_deleted`字段，支持数据恢复
2. **审计字段**：所有表包含`create_time`, `update_time`, `create_by`, `update_by`
3. **JSON字段**：地址、图片等使用JSON类型存储
4. **索引策略**：查询频繁字段建立组合索引
5. **分表考虑**：订单表按时间分表，支持大数据量

**本次权限改造补充约束：**
1. 在现有表结构下，商品的商家归属先通过 `product.create_by` 表达。
2. 订单未显式存储 `merchant_id` 时，通过 `order_item.product_id -> product.create_by` 推导商家可见范围。
3. 普通用户订单范围固定为 `order.user_id = 当前用户ID`。
4. 后续若进入多商户正式场景，应补充 `merchant_id`、`shop_id` 等显式归属字段，而不是长期依赖审计字段。

#### 3.1.3 API设计规范
**统一响应格式：**
```json
{
  "code": 200,
  "message": "success",
  "data": {},  // 业务数据
  "timestamp": 1680556800000
}
```

**分页响应格式：**
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "list": [...],  // 数据列表
    "total": 100,   // 总记录数
    "page": 1,      // 当前页码
    "size": 10,     // 每页大小
    "pages": 10     // 总页数
  }
}
```

#### 3.1.4 角色化接口分层

为避免“后台管理接口”和“用户自助接口”语义混乱，接口分层如下：

1. 后台管理接口
   - `/api/admin/products/**`
   - `/api/admin/orders/**`
   - `/api/admin/users/**`
   - `/api/admin/roles/**`
2. 用户自助接口
   - `/api/products/**`
   - `/api/orders/**`
3. 认证接口
   - `/api/auth/**`

约束规则：
- 后台管理接口默认需要管理员或商家角色，再由服务层做数据范围裁剪。
- 普通用户相关操作统一走 `/api/products/**` 和 `/api/orders/**`。
- 前端隐藏按钮不能替代后端鉴权，服务层必须重复校验。

**错误码规范：**
- `200`：成功
- `400`：客户端错误（参数验证失败）
- `401`：未认证
- `403`：未授权
- `404`：资源不存在
- `500`：服务器内部错误

#### 3.3.2 安全防护措施

3. **CSRF防护**：JWT + CORS配置 + 请求头验证
4. **密码安全**：BCrypt加密 + 盐值
5. **接口限流**：Guava RateLimiter或Redis限流
6. **日志审计**：完整操作日志记录

volumes:
